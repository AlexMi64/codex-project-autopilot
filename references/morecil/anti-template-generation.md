# Morecil Anti-Template Generation

Этот документ задаёт правила, которые помогают плагину делать проекты менее шаблонными.

## Не генерируй “средний проект”

Если описание проекта короткое, не заполняй пустоты самыми типичными решениями.

Вместо этого:

- сначала назови главный сценарий
- потом выдели продуктовый характер
- потом выбери стек и визуальное направление

## Не повторяй одни и те же сигналы

Признаки generic generation:

- одна и та же структура hero/features/testimonials/contact
- одинаковые палитры
- одинаковые шрифтовые решения
- одинаковые объяснения архитектуры
- auth, dashboard, база, billing “на всякий случай”

## Уникальность должна быть управляемой

Нельзя просто просить “сделай необычно”.

Нужно фиксировать:

- аудиторию
- контекст продукта
- характер интерфейса
- полезные ограничения
- anti-generic запреты

## Правило трёх направлений

Перед финальным выбором полезно рассматривать хотя бы три направления:

- безопасное
- рекомендуемое
- более смелое

Но в реализацию идёт только одно.

## Уникальность без хаоса

Не всякая необычность полезна.

Хорошая уникальность:

- усиливает доверие
- делает структуру яснее
- помогает отличать продукт
- поддерживает основной сценарий

Плохая уникальность:

- мешает чтению
- ломает mobile
- усложняет MVP
- добавляет шум вместо характера
